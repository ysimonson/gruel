# Gruel Compiler Benchmarks

This directory contains benchmark programs for measuring Gruel compiler performance.

## Structure

```
benchmarks/
├── manifest.toml       # Benchmark metadata and configuration
├── README.md           # This file
└── stress/             # Stress test programs
    ├── many_functions.gruel    # 1000 functions
    ├── deep_nesting.gruel      # 150 functions with deep nesting
    ├── large_structs.gruel     # 700 struct types
    ├── arithmetic_heavy.gruel  # 250 functions with long arithmetic chains
    ├── control_flow.gruel      # 390 functions with complex control flow
    ├── array_heavy.gruel       # 200 functions with array operations
    ├── register_pressure.gruel # 210 functions with many live variables
    └── comptime_heavy.gruel    # 68 comptime blocks exercising the interpreter
```

## Benchmark Descriptions

| Benchmark | What it tests | Size |
|-----------|--------------|------|
| `many_functions` | Function handling, symbol resolution | 1000 functions |
| `deep_nesting` | Scope handling, block nesting (up to 40 levels) | 150 functions |
| `large_structs` | Type definitions, field access | 700 struct types |
| `arithmetic_heavy` | Expression parsing, codegen | 250 functions, 100+ terms each |
| `control_flow` | CFG construction (if/while/match) | 390 functions |
| `array_heavy` | Array declarations, indexing, modifications | 200 functions |
| `register_pressure` | Register allocation with many live variables | 210 functions |
| `comptime_heavy` | Comptime interpreter (loops, recursion, structs, enums) | 68 comptime blocks |

## Configuration

The `manifest.toml` file controls which benchmarks run and at which optimization levels:

```toml
[config]
opt_levels = ["O0", "O3"]  # Run each benchmark at -O0 and -O3

[[benchmark]]
name = "many_functions"
path = "stress/many_functions.gruel"
description = "1000 functions to stress function handling"
```

The `opt_levels` field specifies which LLVM optimization levels to benchmark. Each
benchmark is compiled and measured at each level, producing results like
`many_functions@O0` and `many_functions@O3`.

## Running Benchmarks

### Using the Benchmark Runner (Recommended)

The `bench.sh` script handles the complete benchmark workflow:

```bash
# Run all benchmarks with defaults (5 iterations, append to history)
./bench.sh

# Custom number of iterations for more accuracy
./bench.sh --iterations 10

# Save to a specific file without updating history
./bench.sh --no-history --output results.json

# Show help
./bench.sh --help
```

The benchmark runner:
1. Builds the compiler in release mode
2. Parses `manifest.toml` to find benchmarks and optimization levels
3. Runs each benchmark at each optimization level, multiple times
4. Measures both compilation time and runtime of compiled binaries
5. Calculates mean and standard deviation for all measurements
6. Outputs JSON results with per-pass timing, runtime, and binary size
7. Appends to history for the performance dashboard

### Running Individual Benchmarks

For manual testing or debugging:

```bash
# Run a single benchmark with timing output
cargo run -p gruel -- --time-passes benchmarks/stress/many_functions.gruel /tmp/out

# Get JSON timing output
cargo run -p gruel -- --benchmark-json benchmarks/stress/many_functions.gruel /tmp/out

# Compile at a specific optimization level
cargo run -p gruel -- --benchmark-json -O3 benchmarks/stress/arithmetic_heavy.gruel /tmp/out
```

## Adding Benchmarks

1. Add a `.gruel` file to `stress/` (or create a new category directory)
2. Add an entry to `manifest.toml`
3. Ensure the program compiles and runs correctly at all configured optimization levels

Each benchmark should:
- Be large enough to produce measurable timing (aim for 1-2 seconds compilation)
- Focus on a specific compiler phase or feature
- Do deterministic computation (no I/O, no randomness) for reliable runtime measurement
- Return a deterministic exit code for verification

## Benchmark History

Results are stored on the perf branch and rendered on the performance dashboard.
Charts are generated by `scripts/generate-charts.py` and include compilation time,
runtime performance, memory usage, binary size, and per-pass breakdowns at each
optimization level.

For more details on the performance tracking workflow, see `docs/perf-branch.md`.
